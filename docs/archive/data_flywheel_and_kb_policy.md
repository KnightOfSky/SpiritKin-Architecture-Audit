## SpiritKinAI 数据闭环与知识库策略（当前建议版）

### 1. 总原则
不要把所有数据都塞进知识库，也不要一上来就主攻训练。

当前推荐顺序：

1. 先把系统主链路跑稳
2. 再把知识库接稳
3. 再把日志与评测闭环做起来
4. 最后再做 SFT / LoRA / 偏好优化

### 2. 四类数据必须分开

#### 2.1 Knowledge（长期稳定知识）
适合放进知识库：
- README
- `docs/` 文档
- 说明书 / 手册 / SOP
- 开发记录与设计决策
- 常见故障排查文档

特点：
- 相对稳定
- 可重复复用
- 能被检索后注入回答

#### 2.2 State（实时状态）
不应直接放进知识库：
- 当前屏幕内容
- 当前窗口 / 当前应用
- 当前节点在线状态
- 当前执行进度
- 当前会话上下文

特点：
- 强时效性
- 过一会儿就失效
- 更适合放在 runtime / session / cache

#### 2.3 Logs（交互与执行日志）
先记录，不要直接全量进知识库：
- 用户输入
- planner 路由结果
- tool 调用
- executor 结果
- 最终回复
- 用户纠正 / 人工反馈

特点：
- 适合分析、回放、排错、构建评测集
- 不等于长期知识

#### 2.4 Training Data（训练数据）
只放经过筛选和标注的数据：
- 高质量问答样本
- 稳定的工具调用轨迹
- 成功 / 失败对照样本
- 人工认可的多轮交互样本

特点：
- 少而准
- 有标签和评分
- 不直接等于原始日志

### 3. 什么应该进入知识库

#### 应该进
- 项目文档
- 架构说明
- 接口文档
- 设备文档
- 业务 SOP
- 售后 / 客服标准话术
- 经确认有效的操作经验

#### 不应该直接进
- 临时聊天记录
- 一次性的屏幕 OCR 噪声
- 实时状态流
- 未确认真伪的自动总结
- 敏感信息 / token / 密钥

### 4. 垃圾数据不是简单删除，而是分层处理

#### 需要清理的典型对象
- 重复文档
- 空文本 / 乱码
- 无意义 OCR 片段
- 过期版本说明
- 错误标注样本
- 含敏感信息的数据

#### 更好的处理方式
- 原始日志保留可追溯副本
- 知识库只保留稳定版本
- 训练集只保留高质量样本
- 低置信度数据进入隔离区等待复核

### 5. 推荐的数据流转闭环

1. **产生数据**
   - 用户对话、工具调用、执行结果、文档更新
2. **分流**
   - 稳定资料 -> knowledge
   - 实时上下文 -> state
   - 交互轨迹 -> logs
   - 高质量样本 -> training_data candidate
3. **清洗与标注**
   - 去重、去敏、纠错、标记来源、标记时间
4. **评测**
   - 看检索是否命中、路由是否稳定、工具调用是否成功
5. **再决定是否训练**
   - 只有评测清楚、数据质量达标时才进入训练

### 6. 知识库更新策略建议

#### 文档类
- 推荐：文档变更时增量 ingest
- 至少保留：`source_path`、更新时间、版本标记

#### 说明书 / SOP 类
- 推荐：人工审核后再入库
- 避免把未确认内容直接作为“可靠知识”

#### 实时状态类
- 不进知识库
- 应进入会话上下文、缓存或状态存储

### 7. 训练策略建议

#### 当前不建议优先做
- 从零训练基础模型
- 在没有评测集的情况下盲目微调
- 把所有日志直接拿去训练

#### 当前更建议做
- 先做 `kb.search` + tool routing + evals
- 先积累真实任务样本
- 先记录失败案例与人工修正
- 先建立高频任务 benchmark

### 8. SpiritKinAI 当前阶段的最优先事项

1. 启动时把项目 `docs/` 自动接入知识库
2. 为日志增加结构化记录字段
3. 建立基础评测集
4. 做知识库去重、来源、版本、过期策略
5. 后面再决定是否做 LoRA / SFT

### 9. 一句话判断标准
- **能长期复用的资料**：进知识库
- **只对当前时刻有用的内容**：进状态层
- **用于追溯和分析的轨迹**：进日志层
- **经过筛选可学习的样本**：进训练集

这四层边界清楚，项目后面才不容易越做越乱。

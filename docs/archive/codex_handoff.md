## 2026-06-12 Architecture Decisions

- 2026-06-25 architecture completion map: the current system is not missing a full rewrite, but 9 architecture blocks are still not closed. Treat these as the required completion checklist before calling the Agent OS architecture complete:
  1. Capability Registry / Capability Graph: stable capability ids such as `publish_product`, `customer_reply`, `generate_video`, `run_tests`, and `upload_product`, mapped to workflows, Skills, policies, KB, required workers, and owner Agents. This is the foundation that keeps the Master from reasoning from scratch every time.
  2. Brain Router / Brain Profiles: first-class `BrainRouterDecision`, `brain_profile`, model cards, cost/latency/context/tool/JSON metadata, local-to-cloud escalation, route audit, and tests. Agents should reference profiles, not raw model ids.
  3. Hybrid Planner Pipeline: staged `TaskAnalyzer -> TaskPlanner -> WorkflowPlanner`, complexity/risk/context scoring, optional cloud planning for high-complexity jobs, local execution handoff, and budget/latency accounting.
  4. Agent Capability Container: a real runtime object that resolves role, KB, memory, workflow templates, Skills, tool scopes, state, and brain policy before any LLM call. Built-in Agents must stop being mostly prompt wrappers.
  5. Worker Pool / Execution Runtime: formalize existing executors into a managed Worker layer with capability, health, workspace, permission, queue, audit, update, and rollback metadata. Close the standalone path `Skill -> Tool -> ExecutionRequest -> Worker`, not only AgentCluster execution.
  6. Replaceable Brain / System-trained Capability: keep Knowledge, Workflow, Skill, Policy, evals, and capability graph independent from models; add BrainAdapter, LoRA adapter registry, model replacement tests, and capability benchmarks.
  7. Code/UI Jury and Patch Synthesizer: structured review packages, `JuryReport`, score extraction, PR/UI screenshot review, and one controlled Merge/Patch Synthesizer. Cloud reviewers judge and advise; they do not directly rewrite live code.
  8. Aggregated Runtime State / Avatar Narrator: backend/shared `ForegroundTask` and `AggregatedRuntimeState`, consistent task progress events from Agents/Skills/Workflows/Android/Remote Worker, dominant activity, highest-priority event, progress, emotional signal, and speech hint.
  9. Local model role policy and scheduler benchmarks: explicit 35B-A3B/27B role split policy, MoE/dense/VRAM/quantization metadata, Q4/Q5 profile selection for 16GB VRAM, and tests for JSON validity, tool-call accuracy, workflow-step completeness, and context drift.
- Recommended implementation order: first Capability Registry + Worker Pool, then Brain Router + Agent Capability Container, then Hybrid Planner, then Jury/eval/promotion gates, then Avatar/Narrator polish and local-model split benchmarks. This order prevents UI or model changes from hiding missing execution architecture.

## 2026-06-12 Thirteen Follow-up Conclusions

1. Cloud models can support self-evolution as reviewers, extractors, and proposal generators, but they must not directly modify live code or enable Skills. Current code supports paper/video artifact ingestion, model extraction, candidate Skill generation, cloud training package review gates, and multi-model review; the missing work is the automatic discovery pipeline from web search to committee decision.
2. Cloud migration is feasible but not one-click. The practical path is containerized frontend, command gateway, event bridge, state/config persistence, HTTPS/WSS, secrets management, and Remote Worker execution for risky local-device actions.
3. Each work project should have its own workspace/repo, dependencies, env files, KB, tasks, and runtime state. The desktop must be fixed so the active workspace follows the opened project instead of staying on the launch root.
4. The self-evolution system is a governed skeleton, not complete automation. It has trajectories, failure samples, eval cases, training packages, artifacts, candidate Skills, and review gates; it still needs real data, replay Harness, staged rollout, rollback, metrics, and committee policies.
5. Desktop Git support exists for status, branch, commit, push, and GitHub CLI detection. GitHub account login is still delegated to system Git or `gh auth login`; there is no first-class desktop OAuth/account manager.
6. Web UI and WPF UI are aligned in theme and some management modules, but not fully equivalent. WPF remains the primary console; web fallback needs a parity audit before it can be treated as identical.
7. OpenClaw and atomic operations are integrated at the software protocol level through atomic operation specs, tools, device adapter, executor, events, and unit tests. Real hardware validation and safety interlocks remain required before production use.
8. Compared with OiiOii-style AI animation Agent products, SpiritKin is stronger in desktop governance, multi-Agent orchestration, KB, workflow, remote/device control, and review gates; it lacks the vertical animation-production product layer: asset library, character/scene/storyboard pipeline, style presets, render queue, gallery, sharing, billing, and cloud render operations.
9. `@Agent` chat routing now has a backend plus desktop first slice. `backend/orchestrator/agent_mentions.py` parses `@agent_id` / `@label` mentions; AgentCluster can answer status/work-queue queries such as `@programming 当前工作情况`, and non-status mentions force Planner routing to the named specialist Agent with mention metadata attached to the reply. WPF now has an `@` composer menu that inserts enabled Agents and status-query prompts, plus a composer status chip showing target Agent and route/status intent. Remaining work: inline autocomplete, richer per-Agent live activity, and command verbs such as `@programming run tests` mapped into governed Skills/workflows.
10. The PDD Link Bridge should be generalized into Android Control Bridge. It needs device identity, task queue, signed command intake, screenshot/state panel, generic atomic Android operations, and per-device permission scopes instead of only PDD link intake.
11. Restricted Remote Worker on USB/removable media is a valid direction but must be zero-trust: unique worker identity, signed tasks/packages, no arbitrary shell by default, allowlisted Skills, audit, revocation, and secure pairing. The USB should be a bootstrap/installer, not the long-term secret holder.
12. Use the existing SpiritKin KB/RAG as runtime memory. Obsidian should be an external authoring/source vault connected through import/index/sync, not a mandatory runtime database.
13. Harness is needed for self-evolution replay/eval/regression. Hermes-style messaging is optional until cloud/multi-node deployment. Live2D is not needed for the current product direction.
